# Building Documentation

This document explains how to build LogViewer documentation locally and how it's automatically built on GitHub.

## Documentation System

LogViewer uses **Doxygen** as its unified documentation system that processes:

1. **API Documentation** - Generated from C++ source code comments (`.hpp`, `.cpp`)
2. **User & Developer Guides** - Markdown files (`.md`) in the `docs/` folder
3. **Architecture Documentation** - Special Doxygen files (`.dox`) with diagrams

All documentation is combined into a single, searchable HTML site generated by Doxygen.

## Building Locally

### Prerequisites

```bash
# macOS
brew install doxygen graphviz

# Linux
sudo apt-get install doxygen graphviz

# Windows (with Chocolatey)
choco install doxygen.install graphviz
```

**Note:** Graphviz is required for generating class diagrams and call graphs.

### Build Commands

**Build All Documentation:**
```bash
# From project root
doxygen Doxyfile

# Output will be in docs/html/
# Open docs/html/index.html in your browser
```

**CMake Integration:**
```bash
# Build documentation through CMake (if Doxygen is found)
cmake --build build-debug --target doc
```

The generated documentation includes:
- API reference from C++ source code
- All Markdown guides (README.md, docs/*.md)
- Architecture diagrams
- Class hierarchy and call graphs

## Automatic GitHub Builds

Documentation is automatically built on GitHub Actions when:

- Code is pushed to `main`, `qt`, or `analysis` branches
- Pull requests modify documentation or source files
- Manually triggered via workflow dispatch

### Workflow: `.github/workflows/documentation.yml`

The workflow performs:

1. **Doxygen Build** - Generates complete documentation from:
   - C++ source code comments
   - Markdown files (README.md and docs/*.md)
   - Architecture diagrams (.dox files)
2. **Artifact Upload** - Stores generated docs for download (30-day retention)
3. **GitHub Pages Deployment** - Deploys to `gh-pages` branch (main branch only)

### Viewing Built Documentation

**During Development:**
- Check workflow run artifacts on GitHub Actions tab
- Download `doxygen-documentation` artifact

**Published Documentation (main branch):**
- **Live Site**: https://bula173.github.io/LogViewer/
- README.md serves as the main page
- Navigate using Doxygen's built-in navigation

## Writing Documentation

### Doxygen (API Documentation)

Use Javadoc-style comments in header files:

```cpp
/**
 * @file EventsContainer.hpp
 * @brief Thread-safe container for log events
 * @author LogViewer Contributors
 */

/**
 * @class EventsContainer
 * @brief High-performance container for storing and retrieving log events
 *
 * This class provides thread-safe operations for adding and accessing
 * log events with O(1) access time.
 *
 * @par Thread Safety
 * All methods are thread-safe. Multiple threads can add events
 * concurrently.
 *
 * @see LogEvent, EventsTableModel
 */
class EventsContainer {
public:
    /**
     * @brief Add a new log event (thread-safe)
     * @param event Event to add (moved)
     * @throws std::runtime_error if container is full
     */
    void AddEvent(LogEvent&& event);
};
```

### Markdown (User Documentation)

Markdown files are automatically processed by Doxygen. Follow these conventions:

- Use clear headings (# for title, ## for sections, ### for subsections)
- Include code examples with language hints
- Add platform-specific instructions when needed
- Use tables for comparisons
- Link to other documentation pages using relative paths

**Example:**
````markdown
# Building on macOS

## Prerequisites

Install Qt 6:
```bash
brew install qt@6
```

## Build Commands

```bash
cmake --preset macos-debug
cmake --build --preset macos-debug-build
```

See @ref DEVELOPMENT.md for more details.
````

**Linking between documents:**
- Use `@ref filename.md` to link to other Markdown files
- Use `@ref ClassName` to link to C++ classes
- Doxygen automatically creates cross-references

## MkDocs Configuration

**Note:** This project no longer uses MkDocs. All documentation is generated by Doxygen, which processes both C++ source code and Markdown files into a unified documentation site.

If you need a separate MkDocs build for a specific purpose, you can create your own `mkdocs.yml` configuration, but it's not part of the standard build process.

## Troubleshooting

### Doxygen Issues

**"dot not found" warning:**
```bash
# Install graphviz
brew install graphviz  # macOS
sudo apt-get install graphviz  # Linux
```

**Empty or missing documentation:**
- Check `INPUT` paths in Doxyfile - should include `README.md`, `docs`, and `src`
- Verify source files have Doxygen comments
- Ensure Markdown files are in the `docs/` directory
- Check `FILE_PATTERNS` includes `*.md`
- Look for warnings in Doxygen output

**Markdown not showing:**
- Verify `USE_MDFILE_AS_MAINPAGE = README.md` is set in Doxyfile
- Check that `FILE_PATTERNS` includes `*.md`
- Ensure `docs` folder is in `INPUT` list
- Markdown files should use standard syntax (Doxygen supports most features)

### MkDocs Issues

**MkDocs is no longer used** - All documentation is now handled by Doxygen.

## Contributing to Documentation

1. **API Documentation**: Add/update Doxygen comments in `.hpp` files
2. **User Guides**: Edit/create `.md` files in `docs/` folder
3. **Main Page**: Edit `README.md` (serves as Doxygen landing page)
4. **Architecture**: Update `.dox` files in `src/dox/` for diagrams
5. **Test Locally**: Build and verify documentation before committing (`doxygen Doxyfile`)
6. **Submit PR**: Documentation changes trigger automatic builds

### Documentation Review Checklist

- [ ] Doxygen comments added for new public APIs
- [ ] Code examples tested and working
- [ ] Links are valid and not broken (use `@ref` for cross-references)
- [ ] Markdown files use proper headings and formatting
- [ ] Screenshots updated if UI changed
- [ ] Spelling and grammar checked
- [ ] Platform-specific instructions included where needed
- [ ] Local build successful (`doxygen Doxyfile`)
- [ ] README.md updated if project structure changed
